# Git Flow Guide

## Основные типы коммитов

| Тип | Назначение |
|------|-------------|
| **feat** | добавление нового функционала |
| **fix** | исправление ошибок |
| **refactor** | изменение структуры или оптимизация кода без изменения логики |
| **chore** | технические задачи (обновление зависимостей, конфигураций и т.п.) |
| **docs** | изменения в документации |
| **style** | правки форматирования, не влияющие на логику |
| **test** | добавление или обновление тестов |
| **perf** | улучшение производительности |

---

## Структура веток

Каждая ветка создаётся под конкретную задачу.

**Формат имени ветки:**

```
<type>/<TASK_KEY>-<краткое-описание>
```

**Примеры:**

```
feat/NAI-123
fix/NAI-456
refactor/NAI-789
```

`TASK_KEY` соответствует ключу задачи из проекта (например, `NAI-123`).

---

## Основной поток разработки

- `main` — стабильная ветка (релизы)
- `develop` — основная ветка разработки
- от `develop` создаются рабочие ветки (`feature/*`, `fix/*`, `refactor/*` и т.д.)

---

## Пример цикла работы

1. Создай ветку от `develop`:
   ```
   git checkout develop
   git pull
   git checkout -b feat/NAI-123
   ```

2. Выполни изменения и закоммить:
   ```
   git commit -m "feat(NAI-123): added login component"
   ```

3. Отправь ветку и создай Pull Request в `develop`.

---

> Минимум описаний — максимум структуры.  
> Все ветки и коммиты строго следуют схеме, чтобы история проекта оставалась чистой.
